*Part 3 of 4. We have an EKS Auto Mode cluster ([Part 1](/part-1/)) running OpenChoreo ([Part 2](/part-2/)). Everything so far shares the host kernel. Now we add the boundary that makes it safe to run untrusted code — [Kata Containers](https://katacontainers.io) — on bare-metal nodes that scale to **zero** so you don't pay for hardware isolation when nothing needs it.*

A container is a process with fences. It shares one kernel with everything else on the node. For an AI agent that can be talked into running arbitrary commands, "shares the kernel with your secrets and your other tenants" is not a boundary you want. **Kata runs each pod inside a lightweight VM with its own kernel.** The catch: Kata needs `/dev/kvm`, and on EC2 that means **bare metal** (~$2/hr and up). The trick that makes it affordable: keep the bare-metal node group at **zero** and let it scale up only when an agent is actually scheduled.

> **Conventions recap:** `<PLACEHOLDERS>`, `export`ed `$VARS`, **✅ Verify** gates, **🐛 gotcha** notes. Re-export Part 1's vars and re-auth if this is a new shell.

## What you'll have at the end of Part 3
- A `c6g.metal` (arm64) **managed node group** coexisting with Auto Mode, **min=0 / max=1 / desired=0**.
- A `kata-qemu` **RuntimeClass** that auto-pins pods to that node group.
- **Cluster Autoscaler** scaling the metal node from **0 → 1** on demand and back to 0 when idle.
- Proof of VM isolation (a different kernel inside the pod).

export AWS_REGION=us-east-1

## Step 1 — A proper node IAM role

Auto Mode's node role is trimmed down; a standard self-bootstrapping node needs the full set. EKS auto-creates the `EC2_LINUX` access entry when the managed node group is created, so we only create the role:

```bash
aws iam create-role --role-name KataNodeRole \
  --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"ec2.amazonaws.com"},"Action":"sts:AssumeRole"}]}'
for P in AmazonEKSWorkerNodePolicy AmazonEKS_CNI_Policy AmazonEC2ContainerRegistryReadOnly; do
  aws iam attach-role-policy --role-name KataNodeRole --policy-arn arn:aws:iam::aws:policy/$P
done
export KATA_NODE_ROLE_ARN=arn:aws:iam::${ACCOUNT_ID}:role/KataNodeRole
```

## Step 2 — Networking for standard nodes (the Auto Mode coexistence trick 🐛)

> 🐛 Auto Mode runs the CNI, kube-proxy, and CoreDNS **internally** — `kubectl get ds -n kube-system` shows almost nothing. Your standard Kata nodes get **no pod networking** unless you install `vpc-cni` and `kube-proxy` — but scoped so they **do not** run on the Auto Mode nodes (which already have their own).

```bash
AFF='{"affinity":{"nodeAffinity":{"requiredDuringSchedulingIgnoredDuringExecution":{"nodeSelectorTerms":[{"matchExpressions":[{"key":"eks.amazonaws.com/compute-type","operator":"NotIn","values":["auto"]}]}]}}}}'

# vpc-cni accepts an affinity override
aws eks create-addon --region $AWS_REGION --cluster-name $CLUSTER_NAME \
  --addon-name vpc-cni --resolve-conflicts OVERWRITE --configuration-values "$AFF"

# kube-proxy does NOT accept affinity in its schema — but AWS ships it pre-scoped
# (its DaemonSet already excludes compute-type in [fargate, auto]). Just install it:
aws eks create-addon --region $AWS_REGION --cluster-name $CLUSTER_NAME \
  --addon-name kube-proxy --resolve-conflicts OVERWRITE
```

```bash
# ✅ Verify both DaemonSets exist with 0 desired pods (no non-auto nodes yet) and won't touch auto nodes
kubectl -n kube-system get ds aws-node kube-proxy -o custom-columns=NAME:.metadata.name,DESIRED:.status.desiredNumberScheduled
# Expected: both DESIRED = 0
```

> **CoreDNS is deliberately skipped here.** A trivial Kata pod needs no cluster DNS, and installing CoreDNS on an Auto Mode cluster is fiddly. We pay this debt in Part 4 (the agent needs *external* DNS) with a one-line pod fix — remember this.

## Step 3 — The bare-metal node group (scale-to-zero)

```bash
# discover private subnets (NAT egress) created by eksctl in Part 1
SUBNETS=$(aws ec2 describe-subnets --region $AWS_REGION \
  --filters "Name=tag:kubernetes.io/role/internal-elb,Values=1" \
            "Name=vpc-id,Values=$(aws eks describe-cluster --name $CLUSTER_NAME --region $AWS_REGION --query 'cluster.resourcesVpcConfig.vpcId' --output text)" \
  --query 'Subnets[].SubnetId' --output text | tr '\t' ' ')
echo "subnets: $SUBNETS"

aws eks create-nodegroup --region $AWS_REGION --cluster-name $CLUSTER_NAME \
  --nodegroup-name kata-nodes \
  --node-role $KATA_NODE_ROLE_ARN \
  --subnets $SUBNETS \
  --instance-types c6g.metal \
  --ami-type AL2023_ARM_64_STANDARD \
  --scaling-config minSize=0,maxSize=1,desiredSize=0 \
  --disk-size 100 \
  --labels kata-enabled=true \
  --taints 'key=sandbox,value=true,effect=NO_SCHEDULE'

aws eks wait nodegroup-active --region $AWS_REGION --cluster-name $CLUSTER_NAME --nodegroup-name kata-nodes
```

**Why these choices:** `c6g.metal` (arm64 Graviton2) is the cheapest reliable bare metal with KVM (~$2.18/hr) — *your workload images must be multi-arch/arm64.* The **taint** repels ordinary pods; the **label** attracts kata-deploy and the RuntimeClass. **`--disk-size 100`** matters: Kata VM images + container images fill the default 20 GB and evict pods. **`maxSize=1`** is a deliberate cost + correctness cap (see the gotcha box).

## Step 4 — Scale-from-zero tags on the ASG 🐛

> 🐛 With the node group at 0, Cluster Autoscaler can't read a live node to learn its labels/taints — it reads them from **ASG tags**. EKS adds the *discovery* tags but **not** the node-template label/taint tags, so CA won't scale up for a pending Kata pod. Add them:

```bash
ASG=$(aws eks describe-nodegroup --region $AWS_REGION --cluster-name $CLUSTER_NAME \
  --nodegroup-name kata-nodes --query 'nodegroup.resources.autoScalingGroups[0].name' --output text)

aws autoscaling create-or-update-tags --tags \
  "ResourceId=$ASG,ResourceType=auto-scaling-group,Key=k8s.io/cluster-autoscaler/node-template/label/kata-enabled,Value=true,PropagateAtLaunch=true" \
  "ResourceId=$ASG,ResourceType=auto-scaling-group,Key=k8s.io/cluster-autoscaler/node-template/label/katacontainers.io/kata-runtime,Value=true,PropagateAtLaunch=false" \
  "ResourceId=$ASG,ResourceType=auto-scaling-group,Key=k8s.io/cluster-autoscaler/node-template/taint/sandbox,Value=true:NoSchedule,PropagateAtLaunch=true"
```

## Step 5 — Cluster Autoscaler, scoped to only the Kata ASG

Auto Mode's own Karpenter won't manage your ASG, so we run Cluster Autoscaler for **just this one ASG**, with IAM via **EKS Pod Identity** (works natively for pods on Auto Mode nodes):

```bash
# IAM role for CA (Pod Identity trust) + scoped policy
aws iam create-role --role-name KataClusterAutoscalerRole \
  --assume-role-policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"pods.eks.amazonaws.com"},"Action":["sts:AssumeRole","sts:TagSession"]}]}'
aws iam put-role-policy --role-name KataClusterAutoscalerRole --policy-name ca --policy-document "{\"Version\":\"2012-10-17\",\"Statement\":[{\"Effect\":\"Allow\",\"Action\":[\"autoscaling:DescribeAutoScalingGroups\",\"autoscaling:DescribeAutoScalingInstances\",\"autoscaling:DescribeLaunchConfigurations\",\"autoscaling:DescribeScalingActivities\",\"autoscaling:DescribeTags\",\"ec2:DescribeImages\",\"ec2:DescribeInstanceTypes\",\"ec2:DescribeLaunchTemplateVersions\",\"ec2:GetInstanceTypesFromInstanceRequirements\",\"eks:DescribeNodegroup\"],\"Resource\":\"*\"},{\"Effect\":\"Allow\",\"Action\":[\"autoscaling:SetDesiredCapacity\",\"autoscaling:TerminateInstanceInAutoScalingGroup\"],\"Resource\":\"*\",\"Condition\":{\"StringEquals\":{\"aws:ResourceTag/k8s.io/cluster-autoscaler/${CLUSTER_NAME}\":\"owned\"}}}]}"

aws eks create-pod-identity-association --region $AWS_REGION --cluster-name $CLUSTER_NAME \
  --namespace kube-system --service-account cluster-autoscaler \
  --role-arn arn:aws:iam::${ACCOUNT_ID}:role/KataClusterAutoscalerRole

# deploy CA, bound explicitly to the kata ASG
helm repo add autoscaler https://kubernetes.github.io/autoscaler >/dev/null 2>&1; helm repo update autoscaler >/dev/null
helm upgrade --install cluster-autoscaler autoscaler/cluster-autoscaler -n kube-system --wait --values - <<EOF
awsRegion: ${AWS_REGION}
cloudProvider: aws
rbac: { serviceAccount: { create: true, name: cluster-autoscaler } }
autoscalingGroups:
  - { name: ${ASG}, minSize: 0, maxSize: 1 }
image: { tag: v1.32.0 }
extraArgs:
  scale-down-unneeded-time: 10m
  max-node-provision-time: 15m
  balance-similar-node-groups: "false"
EOF
```

```bash
# ✅ Verify CA authenticated (Pod Identity) and sees the ASG — no AccessDenied
kubectl -n kube-system logs deploy/cluster-autoscaler-aws-cluster-autoscaler --tail=30 | grep -i "$ASG" | head -2
```

## Step 6 — Install Kata + the RuntimeClass

```bash
KATA_VER=$(curl -fsSL https://api.github.com/repos/kata-containers/kata-containers/releases/latest | python3 -c 'import sys,json;print(json.load(sys.stdin)["tag_name"])')

# kata-deploy DaemonSet, scoped to the Kata nodes + tolerating the taint, qemu shim only
helm upgrade --install kata-deploy oci://ghcr.io/kata-containers/kata-deploy-charts/kata-deploy \
  --version "$KATA_VER" -n kube-system --wait --values - <<'EOF'
deploymentMode: daemonset
nodeSelector: { kata-enabled: "true" }
tolerations: [ { key: sandbox, operator: Equal, value: "true", effect: NoSchedule } ]
shims: { disableAll: true, qemu: { enabled: true, supportedArches: [arm64], runtimeClass: { nodeSelector: { kata-enabled: "true" } } } }
runtimeClasses: { enabled: true, createDefault: false }
node-feature-discovery: { enabled: false }
EOF
# (GHCR 403? run: helm registry login ghcr.io  with a GitHub PAT, then retry)

# the chart's RuntimeClass gets a nodeSelector but no toleration — patch it in
kubectl patch runtimeclass kata-qemu --type merge -p \
  '{"scheduling":{"tolerations":[{"key":"sandbox","operator":"Equal","value":"true","effect":"NoSchedule"}]}}'
```

```bash
# ✅ Verify the RuntimeClass has BOTH nodeSelector and tolerations
kubectl get runtimeclass kata-qemu -o jsonpath='{.scheduling}'; echo
```

> **Why the patch matters:** a pod that merely *tolerates* a taint is not *pinned* to that node. The RuntimeClass needs a `nodeSelector` (to attract) **and** a `toleration` (to pass the taint). Set `runtimeClassName: kata-qemu` on a pod and it lands on a Kata node automatically.

## Step 7 — Prove it (and prove scale-to-zero)

```bash
kubectl run kata-test --image=nginx:alpine --overrides='{"spec":{"runtimeClassName":"kata-qemu"}}'
kubectl wait --for=condition=ready pod/kata-test --timeout=420s     # ~2-3 min: metal boot + kata-deploy
NODE=$(kubectl get pod kata-test -o jsonpath='{.spec.nodeName}')

echo "guest kernel: $(kubectl exec kata-test -- uname -r)"
echo "host kernel:  $(kubectl get node $NODE -o jsonpath='{.status.nodeInfo.kernelVersion}')"
kubectl exec kata-test -- sh -c 'nproc; grep MemTotal /proc/meminfo'
```

**✅ Verify:** the guest kernel (e.g. `6.18.x`) differs from the host kernel (e.g. `6.12.x-amzn2023`), and the pod sees ~1 CPU / ~2 GB — not the host's 64 vCPU / 128 GB. That's a real VM.

```bash
kubectl delete pod kata-test        # idle -> CA scales the metal node back to 0 in ~10 min -> $0 at rest
```

## 🐛 What bit me

- **CA wouldn't scale from zero** — EKS added only the *discovery* tags, not the node-template label/taint tags (Step 4). Without them CA can't tell a scaled node would satisfy the pending pod.
- **CA over-provisioned a second $2/hr metal node** for one pod. The pod waits on the `katacontainers.io/kata-runtime` label that kata-deploy adds *after* the node boots, so CA briefly thinks the first node doesn't qualify and adds another. Capping **`maxSize=1`** fixes it (one 64-vCPU box is plenty) and hard-caps cost.
- **arm64 kernel strings are both `aarch64`** — the proof is the *version* gap plus `virtiofs`/`kataShared` mounts, not the architecture. Don't promise the audience a different arch.
- **Deferred CoreDNS is a landmine for Part 4** — the agent will need to resolve `api.anthropic.com` from the Kata node. One-line fix, coming up.

## Simpler alternative (if you don't need scale-to-zero)
A static `eksctl` `c5.metal` node group (x86, `desiredCapacity: 1`, no autoscaler, no Auto Mode coexistence) is fewer moving parts — see my earlier Medium article. You trade ~$3k/mo of always-on bare metal for simplicity. This series keeps it at $0-when-idle.
